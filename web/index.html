🔑 Connecting Your Private Infrastructure

Since this hosted dashboard is running on a public cloud server, it operates in Simulated Mode (it has no access to your private local networks). To deploy actual containers on your own local hosts or remote clusters, use one of the following methods:

Option A: Local Binary Daemon (Recommended) Run DoktriAI on your private host. The orchestrator will automatically bind to your local socket: # Run locally in your terminal
go run ./cmd/doktriai-api -addr :18080 Once running, the local dashboard at localhost:18080 will read and deploy container processes directly to your local Docker daemon socket.
Option B: Remote SSH / TCP Tunneling Expose your private Docker daemon securely via SSH or TLS TCP sockets, and configure your target environment settings: # Override target Docker Host URL
export DOCKER_HOST="tcp://your-private-ip:2376" Attach client TLS certificates in ~/.docker/ or specify SSH keys so the cloud api can invoke container workloads remotely.

1. Platform Architecture

DOKTRIAI is an autonomous, declarative control plane designed to manage, self-heal, and scale containerized workloads. The control plane relies on a continuous Reconciliation Loop written in Go that constantly compares the target state (stored in sqlite) against the actual runtime state (Docker daemon or Kubernetes cluster via kubectl APIs).

💡 Control Plane Lock Mutex You can dynamically freeze the reconciler loop by acquiring the global lock (via POST /api/lock or the Reconciler panel). This stops self-healing temporarily so engineers can perform direct runtime debugging without automated rollbacks.

2. Declarative Schema Keywords

Workloads are declared using a standard structured YAML manifest. Below is the specification reference for each keyword:

Keyword Type Meaning & Valid Values
apiVersion string Specifies API schema version. Use doktri.io/v1alpha1.
kind string Resource type descriptor. Must be Workload.
metadata.name string Workload identity name. Lowercase alphanumeric, e.g. nginx-web.
metadata.project string Grouping namespace. Matches active sidebar explorer projects.
spec.image string Docker image tag, e.g. nginx:alpine or redis:latest.
spec.replicas integer Desired running containers count. Allowed bounds: 1 to 50.
spec.deployStrategy string Deployment style: rolling (gradual replacement) or recreate.
spec.ports array List of maps matching public hostPort to internal containerPort.

3. OPA RBAC & PTE Security Gate

DoktriAI implements policy-as-code guardrails via the Open Policy Agent (OPA) engine:

  • DevOps Lead (Admin): Full operational rights. Can deploy instantly, scale up to 50 replicas, and manually approve plans.
  • Junior SRE (Operator): Basic deployment rights. Attempts to delete workloads or scale replicas > 5 automatically triggers the Plan-To-Execute (PTE) Gate. The reconciler stalls changes until an Admin authorizes the plan.
  • Security Auditor (Reader): Read-only context. Any deployment write attempts are blocked at the router gateway level.
  • Behavioral Anomaly Engine: Monitors actor rates in a 5-minute sliding window. Exceeding thresholds flags the actor and temporarily suspends their token validity.

4. AI Agent & MCP Integration

AI Agents (Claude, GPT, Gemini) call control plane actions as standard tool invocations using the Model Context Protocol (MCP) served at POST /api/mcp.

Exposed MCP tool bindings:

  • tools/list: Returns list of schema tool specs.
  • deploy_workload: Creates/updates container workload spec.
  • list_workloads: Queries running instances status list.
  • delete_workload: Removes a workload spec from target provider.
  • get_logs: Retrieves streaming logs from active containers.
👁 Cryptographic JWT and HMAC Keys All agent transactions must include a valid signed JWT key in the X-Doktri-Token header. Generate tokens using the generator in the CLI Reference workspace panel.

        "image":         "nginx:alpine",

Control Plane Execution Flow Guide

⓵ Input Trigger

Developer or AI Agent pushes deployment intent via Web UI (YAML Editor), CLI command, or MCP JSON-RPC protocol.

⓶ Gateway & Auth

doktriai-api interceptor validates signatures, verifies stateless JWT claims, and logs actions to the secure audit trail.

⓷ State Reconciler

Core engine validates schema and screens spec against doktri-policy.yaml rules (registry allowlist, replica thresholds).

⓸ Database Store

Core runs reconciliation loop, persisting version snaps in DB history and detecting drift from desired configuration.

⓹ Runtime Drivers

Virtualization drivers (Docker host socket/Kubernetes operator) execute upgrades (rolling vs recreate strategies) to achieve target state.
